# AI-Assisted Squash-Merge Conflict Resolution

The canonical, always-current merge instructions are **generated by the workflow itself** when
`pnpm wp-start-upsert-pr` (or `pnpm wp-start-update`) hits conflicts. They are written to:

```
.webpieces/instruct-ai/webpieces.mergeprocess.md
```

That generated file is parameterized with the live merge directory and the actual conflicted-file
list, so it can never drift from the real layout. **Read it and follow it exactly.**

## What to do

1. Read `.webpieces/instruct-ai/webpieces.mergeprocess.md` and follow its steps:
   resolve each conflicted file from the 3-point context (A=fork, B=feature, C=main), write a
   `merge-explanation.md` next to each file's diffs explaining how you merged it, then run the
   FINISH command that PAIRS with the start command you ran:
   - started with `pnpm wp-start-update` (no PR open) → finish with `pnpm wp-finish-update`
   - started with `pnpm wp-start-upsert-pr` (a PR is open) → finish with `pnpm wp-finish-upsert-pr`
     (it validates the merge, requires your review.json, and creates/updates the PR)

   Never mix the halves — `wp-start-update` + `wp-finish-upsert-pr` is not a flow. The generated
   `webpieces.mergeprocess.md` names the exact finish command for the run you are in.
2. If that file does **not** exist, there is no merge in progress — run `pnpm wp-start-update` (no PR
   open) or `pnpm wp-start-upsert-pr` (a PR IS open — required then, since the 3-point merge rewrites
   the branch and the PR must be re-pointed in the same run) to start (or re-start) the squash-merge
   from main; it will generate the file if conflicts occur.

Do NOT run `git add` / `git commit` / `git push` / `gh pr` yourself while a merge is in progress —
the `merge-in-progress-guard` hook blocks them until your flow's finish command validates.
